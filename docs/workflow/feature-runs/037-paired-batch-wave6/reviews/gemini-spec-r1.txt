## Important findings (gaps in requirements coverage)

1.  **File:** `docs/tech-debt/wave6-spec.md`
    **Section:** Task 1 — Build the replacement card
    **Finding:** The spec does not address how the new card should behave for **aggregate runs**. Aggregate runs are a known concept in the project, but the spec is silent on whether the card should render for them. If a user views an aggregate run based on a paired vignette, the card's visibility rule (`hasMirroredValueTokens`) would be met, and it would render, potentially showing confusing "pooled-of-pooled" metrics without context.
    **Recommendation:** The spec must explicitly define the card's behavior for aggregate runs. Options include: (a) hiding the card entirely for aggregate runs, or (b) showing the card but with a clear disclaimer in the header explaining that the metrics are pooled from aggregate data.

2.  **File:** `docs/tech-debt/wave6-spec.md`
    **Section:** Task 1 — Build the replacement card -> Per-model table
    **Finding:** The spec's guidance for null `pressureResponse` is incomplete. It says to "Render `—` when null." However, the `pressureSensitivity` query provides a `reason` field explaining *why* it might be null (e.g., `'directional-thin'`). By ignoring this, the card fails to distinguish between "not enough data to compute" and a potential "computed but zero-effect" result, which is a meaningful distinction for a researcher.
    **Recommendation:** Update the spec to require the card to use the `pressureResponse.reason` field. When `pressureResponse.value` is null, the `—` in the UI should have a tooltip explaining the reason, derived from the `reason` field.

3.  **File:** `docs/tech-debt/wave6-spec.md`

However, as noted in the critical findings, this is not protected by a specified test.
2.  **Null handling:** As noted in important findings, the spec's handling of null pressure response is too simplistic and misses an opportunity to convey important methodological context (thin data) to the user.
3.  **Aggregate runs:** The spec's failure to consider aggregate runs is a significant methodology gap, as it could lead to the display of confusing or misleading metrics without proper context.
4.  **Directions measured:** The spec correctly handles the "directions measured" concern by keying off the signature. The `pressureSensitivity` query is signature-specific, so if a mirrored vignette has no runs *at that signature*, the message "Mirrored runs at signature `vnewtd` will populate..." will correctly be shown. This preserves the methodological integrity.
