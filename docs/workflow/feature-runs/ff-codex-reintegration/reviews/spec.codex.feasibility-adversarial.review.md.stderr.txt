**Feature branch**: `claude/ff-codex-reintegration`

## Summary

Five narrow runner improvements that together restore "Codex implements, Claude orchestrates" as the default flow:

1. **`dispatch-codex` runner subcommand** — wraps `codex exec`, captures stdout/stderr to disk, appends a typed record to `state.codex_dispatches[]`. The missing producer for the field PR #751 added.
2. **Freshness-bound suppression in `check_implementation_rule`** — the implementation-rule WARN suppresses only when a dispatch is an ancestor of HEAD AND its line count is within 50 of current. Stale or unrelated dispatches no longer suppress.
3. **`advance` subcommand** (Fix 3 from `feature-factory-runner-fixes.md`) — manual escape hatch that sets `judge_next_action="advance"` with a recorded reason, for the rare case where the runner gets wedged on manifest drift after the 3-round adversarial cap.
4. **Banner rename `repair_X_checkpoint` → `run_X_checkpoint`** (Fix 4 from `feature-factory-runner-fixes.md`) — pure string change. The word "repair" reads as "something broke"; "run" matches what the command actually does on the happy path.
5. **Better branch-base fallback for the implementation-rule check** — replace `HEAD~50` (silently under-reports on long-lived branches) with `git merge-base --fork-point origin/main HEAD`, then skip the check entirely with a printed note if even that fails. Honest skip beats silent under-report.

None of these changes are user-facing in the product; all of them are runner-facing for the FF workflow.

## Problem statement

### Fix 1 — `dispatch-codex` runner subcommand

PR #751 added `factory_deliver.check_implementation_rule`, which counts non-test lines added on the branch and emits a WARN above 200 lines. The check suppresses when `state.codex_dispatches[]` is non-empty. The intent: if Codex did the work, the rule doesn't apply; if Claude did the work, surface it at deliver time.

The producer side is missing. There's no runner command that writes to `codex_dispatches[]`. Today every feature that goes through Codex would still trip the WARN, because nothing records the dispatch. Operators currently dispatch Codex by running `codex exec` directly from the shell — outside the runner's view.

This feature adds a `dispatch-codex` subcommand that:
- Reads a prompt file the operator prepared.
- Invokes `codex exec -m <model> -s workspace-write` from the repo root.
- Streams stdout/stderr to `docs/workflow/feature-runs/<slug>/codex-dispatches/<UTC-timestamp>/{stdout.txt, stderr.txt}`.
- Appends a structured record to `state.codex_dispatches[]`.
- On Codex quota exhaustion (re-using `is_codex_quota_exhaustion` from PR #751's `run_codex_review.py`), exits 4 with a clear message and does NOT record the failed dispatch (no false suppression).

### Fix 2 — Freshness-bound suppression in `check_implementation_rule`

PR #751's review caught this as an accepted-limitation: today, any non-empty `codex_dispatches[]` suppresses the WARN. That means a stale dispatch from an earlier branch — or even a real dispatch followed by 5,000 lines of post-hoc Claude edits — still suppresses.

The fix: a dispatch entry suppresses the WARN only when:
- `git merge-base --is-ancestor <entry.head_sha> HEAD` succeeds (the dispatch happened on an ancestor of the current branch tip), AND
- The current `_added_code_lines(branch_base)` is within 50 of `entry.lines_added_at_dispatch_time`.

Stale dispatches (head_sha unreachable, or current line count is 50+ beyond the dispatch's recorded count) do NOT suppress. When a fresh dispatch DOES suppress, print a one-line note showing which dispatch covered the check.

### Fix 3 — `advance` subcommand

From `feature-factory-runner-fixes.md` Cause 5: there is no `--force-advance`, `--skip-manifest-check`, or `--accept-judge-verdict` flag in the runner. The judge panel writes `judge_next_action="advance"` to state.json (PR #744 wired the runner's decision tree to honor it), but if the manifest is also unhealthy due to post-judge artifact edits, the state can still feel wedged. Today the only escape is hand-editing state.json + review frontmatter SHAs.

The fix: a small `advance` subcommand that sets `state.stages[stage].judge_next_action = "advance"` and appends `{stage, ts, reason, head_sha}` to `state.annotations[]`. Reason length is eager-validated (≥20 chars) before any state mutation. Combined with PR #744's decision-tree change, the runner moves past the stage on its next evaluation. The `repair` subcommand stays unchanged for actual manifest-repair cases.

### Fix 4 — Banner rename `repair_X_checkpoint` → `run_X_checkpoint`

From `feature-factory-runner-fixes.md` Cause 4: the state machine uses `repair_X_checkpoint` as the name for "this stage needs its checkpoint run." The word "repair" reads to a human (and to an orchestrating AI) as "something broke, fix it" — but mechanically it's just "run the checkpoint command for stage X." Every transition between stages surfaces a `repair_next_checkpoint` banner, making the workflow feel broken even on the happy path.

The fix: pure string change in `factory_next_action.py` and any callers that emit the banner. The `repair` subcommand name stays as-is (it's used for actual manifest-repair). Test expectations updated to match.

### Fix 5 — Better branch-base fallback for the implementation-rule check

PR #751's `_resolve_branch_base()` falls back to `HEAD~50` when `origin/main` and `main` cannot be resolved as merge bases. On a long-lived feature branch with more than 50 commits since it diverged from main, `HEAD~50` lands inside the feature branch — so the diff-line count only reflects the last 50 commits, not the actual branch work. The check silently under-reports.

The fix: new fallback chain:
1. `git merge-base origin/main HEAD`
2. `git merge-base main HEAD`
3. `git merge-base --fork-point origin/main HEAD` (correct branch-point even on long-lived branches)
4. Return `None` and **skip the check entirely** with a printed stderr note: `implementation-rule check skipped — could not resolve branch base (origin/main, main, fork-point all failed)`.

Honest skip is better than a silent under-report.

### Fix 2 — Freshness-bound suppression

- **FR-008**: `factory_deliver.check_implementation_rule(slug)` MUST iterate `state.codex_dispatches[]` and consider an entry "fresh" only when ALL of (a) `entry.exit_code == 0`, AND (b) `git merge-base --is-ancestor <entry.head_sha> HEAD` returns 0, AND (c) `entry.lines_added_at_dispatch_time` is not null AND can be compared to a current snapshot (per FR-008b), AND (d) `abs(<current snapshot> - entry.lines_added_at_dispatch_time) <= 50`.
- **FR-008a**: An entry with `lines_added_at_dispatch_time: null` MUST NOT be treated as fresh on its own. If recomputation succeeds AND the resulting drift is ≤50, the entry MAY suppress. If recomputation fails (e.g., current branch-base also unresolved, or `entry.head_sha` unreachable), the entry stays not-fresh. This avoids permanently locking out dispatches whose branch-base was transiently unresolved at dispatch time.
- **FR-008b**: When `entry.branch_base_sha` is recorded (non-null) AND differs from the current `_resolve_branch_base()` result, the consumer MUST recompute the dispatch's line count as `git diff --numstat <current_branch_base>..<entry.head_sha>` (re-baseline, not stored value), then compare to the current `_added_code_lines(current_branch_base)`. - **FR-009**: When at least one fresh entry is found and the line count is above threshold, System MUST suppress the WARN AND print a one-line note: `implementation-rule check skipped — covered by codex dispatch <ts> (sha <head_sha[:7]>, +<lines>)`. Use the most recent fresh entry for the note.

## Risks

- **R1 (medium)**: Codex's CLI behavior across versions could differ (`-s workspace-write` flag, `-m` flag). **Mitigation**: tests mock `subprocess.run` directly; we do not assert on Codex internals. The dispatch script trusts the operator to have a working Codex CLI.
- **R2 (medium)**: `git merge-base --fork-point` requires reflog data, which can be missing in shallow clones or after reflog expiry. **Mitigation**: skip-on-fail is the documented behavior; the message tells the operator to investigate.
- **R3 (low)**: `lines_added_at_dispatch_time` is a snapshot at dispatch start, before Codex actually runs. So a 0-line dispatch (Codex makes no changes) records `lines_added_at_dispatch_time=0`, and at deliver time the WARN can still fire on subsequent Claude edits. This is intentional — we want post-dispatch Claude edits to be visible. **Mitigation**: documented behavior; the suppression note shows the `+<lines>` count so the operator can sanity-check.
- **R4 (low)**: `prompt_sha256` is computed before Codex modifies anything, so if the operator edits the prompt file mid-run (between read and codex-exec), the recorded sha could be stale relative to what Codex saw. **Mitigation**: read the file once, hash, then pass via `cat` — the same content goes to both. (Implementation note for plan.)
- **R5 (low)**: The `advance` subcommand bypasses the manifest health check by design. A careless reason field could mask a real issue. **Mitigation**: reason ≥20 chars + recorded in `annotations[]` with `head_sha` for audit. This matches the existing `--override-implementation-reason` pattern from PR #751.

## Findings
1. **Medium**: `FR-003` and `FR-004` do not define a single, unambiguous identifier for back-to-back dispatches that land in the same microsecond. The spec says to append `_NNN` to make the path unique, but it also says `ts` is the “same as the directory timestamp” and fixes the timestamp format without the suffix. That leaves `ts` either non-unique or out of spec. This will make audit records hard to round-trip back to the filesystem.

2. **Medium [UNVERIFIED]**: The new `skipped` status in `FR-009a`/`FR-021` can be erased by the proposed boolean compatibility shim. If any caller keeps using truthiness or only checks for `"triggered"`, a branch-base resolution failure will look the same as a clean pass, which recreates the silent-under-report problem the spec is trying to fix. The spec should force status-aware handling at the deliver call site, not just “if needed” guidance.

3. **Medium**: `dispatch-codex` passes the entire prompt as a single argv argument and defines no hard size limit or fallback path. The spec says prompts are usually small, but it does not say what happens when one is large enough to hit the OS argv limit. That makes a core workflow step fail unpredictably on longer prompts, and this command is supposed to be the only producer for `state.codex_dispatches[]`.

## Residual Risks
- `git merge-base --fork-point` is still fragile in shallow clones or after reflog expiry, so the implementation-rule check can still end up in the skipped path on some real repos.
- The `advance` subcommand is a manual override by design. It improves escape velocity, but it also gives operators a way to paper over genuine manifest problems if they use it casually.
- The 50-line freshness window is still a heuristic. Even with the ancestry check, borderline cases can still flip between suppressed and triggered after small post-dispatch edits.
