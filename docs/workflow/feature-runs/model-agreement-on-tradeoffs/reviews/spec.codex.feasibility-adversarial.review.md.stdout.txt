## Findings
- Medium [CODE-CONFIRMED]: The spec says the main query and the drilldown "read from the same snapshot" and that the second query hits "the same snapshot cache," but the current read path does a fresh DB lookup every time. `readValuePairModelVotesFromSnapshot()` calls `getCurrentSnapshot()`, which does a new `findFirst(...)` with no request-scoped memoization or snapshot ID pinning. That means the matrix and drilldown can diverge if a refresh lands between the two requests, and the "one DB read per page load" claim is not supported by the code. See [`domain-analysis-cache.ts`](/Users/chrislaw/valuerank/cloud/apps/api/src/services/analysis/domain-analysis-cache.ts) and [`model-grouping-significance.ts`](/Users/chrislaw/valuerank/cloud/apps/api/src/graphql/queries/model-grouping-significance.ts).
- Medium [CODE-CONFIRMED]: The spec does not define what happens when the new `unavailableModels` filter leaves fewer than two comparable models. The current resolver already requires at least two selected models up front, and it hard-fails when coverage is missing, so the new design needs an explicit fallback for the `< 2 surviving models` case. Without that, the report has no valid pairwise matrix and no safe default pair to open in the drilldown. See [`model-grouping-significance.ts`](/Users/chrislaw/valuerank/cloud/apps/api/src/graphql/queries/model-grouping-significance.ts).
- Medium [UNVERIFIED]: The integration-test anchor for the "A always A, B always B" case is mathematically inconsistent with the spec’s own kappa formula. With `P_observed = 0` and `P_chance = 0`, Cohen's kappa evaluates to `0`, not `-1` or `null`. If that anchor stays as written, it will force the implementation or the test to be wrong.

## Residual Risks
- The spec still makes the equal-weighting rule hard to audit because the output schema does not expose per-vignette sample counts. Users can see `totalCells`, but not how many vignettes contributed to each averaged row.
- The `noisy` badge and the `0.7` / `5` thresholds are explicitly heuristic, but the UI copy has to be very careful or users will read them as statistical guarantees.
- Pending-state behavior is still under-specified for the new fields. `excludedNonBinaryCells`, empty arrays, and null metrics need a consistent contract so the UI does not guess at loading vs empty vs degenerate states.
