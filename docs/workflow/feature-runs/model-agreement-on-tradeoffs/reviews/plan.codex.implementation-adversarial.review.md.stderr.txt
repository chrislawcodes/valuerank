# Plan

## Architecture summary

The feature replaces a Wilcoxon-based pairwise-significance section with a Cohen's-kappa-based agreement section on `/models`. The replacement runs in one PR — both backend and frontend ship together so no stale GraphQL shape is ever served to a deployed client.

**Three layers of work:**

1. **Snapshot data model** — extend the domain-analysis snapshot output to include cell-level outcomes (`cellLevelOutcomes`) alongside the existing `valuePairModelVotes`. Bump the snapshot code version `1.11.0 → 1.12.0` to force rebuilds.
2. **API resolver** — replace `modelGroupingSignificance` with `modelAgreementOnTradeoffs` (matrix + trial consistency) and a separate `modelPairDivergenceBreakdown` (drilldown). New math library for kappa / percent agreement / trial-consistency aggregation.
3. **Web UI** — replace `ModelGroupingSignificanceSection` with `ModelAgreementSection` containing three stacked sub-reports. Reuse the existing heatmap component (renamed and recolored). Delete the table component and significance-specific operations.

**Data flow (no change):**

```
Transcripts → cellMap (5-tuple keys) → snapshot output
                                         ↓
                                  cellLevelOutcomes  →  resolver
                                                          ↓
                                                  Pothos GraphQL
                                                          ↓
                                                       UI
```

The transcript ingestion and `cellMap` accumulation are unchanged. The new code lives in the path from `cellMap` → `cellLevelOutcomes` and downstream of that.

## Architecture decisions

### A1. Two queries vs. one (drilldown)

**Decision:** two top-level queries — `modelAgreementOnTradeoffs` (matrix + consistency) and `modelPairDivergenceBreakdown` (drilldown for one pair).

**Rationale:** the drilldown payload grows as M*(M-1)/2 if nested in every matrix row. With 8+ models that's 28+ pair entries each carrying 6 value-pair breakdowns. A separate query that the UI fires only when a pair is selected keeps the matrix payload lean. Both queries read the same snapshot — the second query is fast because the snapshot is already in DB cache after the first.

**Alternative considered:** a single nested query. Rejected because of the M^2 payload growth (Gemini round-1 H2).

### A2. Cell-level outcomes derived from existing `cellMap`

**Decision:** the new `cellLevelOutcomes` field is computed inside `buildSnapshotOutput` by collapsing `cellMap`'s two valueKey entries per physical cell into a single canonical-direction entry. No changes to `accumulateTranscriptCells`.

**Algorithm — explicit pseudocode (addressing Gemini plan H1):**

```
# cellMap key: definitionId::modelId::valueKey::ownLevel::opponentLevel
# For each physical cell, cellMap has TWO entries — one per valueKey in the pair.
# We want one cellLevelOutcomes entry per physical cell with canonical direction.

groups: Map<(definitionId, modelId, ownLevel, opponentLevel), Map<valueKey, CellCounts>>

for each (key, counts) in cellMap:
    parse key → (defId, modelId, valueKey, ownLevel, oppLevel)
    groupKey = (defId, modelId, ownLevel, oppLevel)
    groups[groupKey][valueKey] = counts

for each (groupKey, byValueKey) in groups:
    if size of byValueKey != 2: skip   # non-binary value pair, see § Non-binary cells
    sortedKeys = sort(keys of byValueKey)   # alphabetical
    canonicalA, canonicalB = sortedKeys[0], sortedKeys[1]
    aCell = byValueKey[canonicalA]
    bCell = byValueKey[canonicalB]
    # In cellMap, "wins" for valueKey X = times the model chose X.
    # So aChoices = aCell.wins (times model chose canonicalA)
    # And bChoices = bCell.wins (times model chose canonicalB)
    # Sanity invariant: aCell.wins == bCell.losses, aCell.losses == bCell.wins
    outKey = `${defId}::${modelId}::${canonicalA}::${canonicalB}::${ownLevel}::${oppLevel}`
    cellLevelOutcomes[outKey] = { aChoices: aCell.wins, bChoices: bCell.wins, neutrals: aCell.neutrals }
```

**Rationale:** `cellMap` already carries `(definitionId, modelId, valueKey, ownLevel, opponentLevel)` keys per `transcript-cell-accumulator.ts:30-32`. The grouping key for the new pass is `(definitionId, modelId, ownLevel, opponentLevel)` (drops `valueKey`) so both value-key perspectives of the same physical cell land in the same group; canonical direction is then determined by sorting valueKeys within the group.

**Why alphabetical-first is safe.** ValueKeys are drawn from `DOMAIN_ANALYSIS_VALUE_KEYS` (`cloud/apps/api/src/graphql/queries/domain-analysis-values.ts`), a fixed const-array of well-formed Schwartz value names (lowercase, no whitespace, no unicode). The TypeScript `Array.prototype.sort()` default lexicographic order is deterministic for these inputs. Kappa and absolute divergence are both **invariant under canonical-direction choice** — kappa is unchanged if you swap A and B labels for both models simultaneously; absolute divergence is symmetric. So the alphabetical convention is purely a labeling choice with no statistical effect.

**Alternative considered:** modify `accumulateTranscriptCells` to write canonical-direction entries directly. Rejected — couples this feature to the upstream accumulator and risks breaking other consumers of `cellMap` (the existing `valuePairModelVotes` and `computeCellWeightedDomainRates` both read it).

### A3. Both fields written by new builder for one release

**Decision:** the new builder writes BOTH `valuePairModelVotes` (legacy) and `cellLevelOutcomes` (new). The legacy two-pass loop is not removed in this PR.

**Rationale:** during the deploy window, an old web client (cached JS bundle) may still call the old `modelGroupingSignificance` resolver; if that resolver is already deleted, the call 404s — that's expected. But if the OLD resolver is briefly still deployed (pre-rollout) and reads a NEW snapshot, it would return empty data without `valuePairModelVotes`. Dual-write removes that gap. The legacy field is removed in a follow-up PR after `1.12.0` snapshots are universal.

**Verification:** unit test asserts both `valuePairModelVotes` and `cellLevelOutcomes` are present on the snapshot output object after `buildSnapshotOutput` runs.

### A4. Tied cells excluded from BOTH kappa and divergence (revised after Codex plan M3)

**Decision:** when either model has `proportionA` within `±epsilon` of `0.5` on a cell (`epsilon = 1e-9` to defeat floating-point drift — Gemini plan M3), that cell is excluded from BOTH kappa and `meanAbsoluteDivergence` for that pair. The cell IS counted in the model's own Trial Consistency calculation (where the tie itself is the data point — a 50/50 cell tells you about model decisiveness).

**Rationale (revised from spec):** the original spec said exclude from kappa, include in divergence. Codex correctly pointed out that this means heatmap and drilldown measure different cell populations and can disagree confusingly. Excluding from both is cleaner; the user gets one consistent denominator across the report.

**Verification:** 
- Unit test: a 3/3 cell (`proportionA == 0.5`) is excluded from kappa AND divergence.
- Unit test: cells with `proportionA = 0.5000000001`, `1/2`, `2/4`, `3/6` all detected as ties via `Math.abs(proportionA - 0.5) < 1e-9`.
- The `excludedTiedCells` count is exposed in the schema (added during reconciliation — see Wave 2). Updates to spec § Output schema also required.

### A5. Trial Consistency naming and caveats

**Decision:** the metric `max(proportionA, 1 − proportionA)` is named "Trial Consistency" (NOT "Run-to-run Reliability" or "Self-consistency"). Hard UI requirements: info-icon tooltip on the column header + footnote below the table explaining what the metric measures and what it doesn't.

**Rationale:** the snapshot aggregates across all trials in scope and discards run identity. Multiple trials of the same cell come from multiple runs in practice, but they could also reflect orientation flips. Calling it "Run-to-run Reliability" overclaims; calling it "Trial Consistency" is honest.

**Verification:** UI snapshot test asserts the tooltip and footnote are rendered. Lint rule (custom or comment-based) flags any rename to "self-consistency" / "reliability."

### A6b. Trial Consistency support requirement (added after Codex plan M2)

**Decision:** a cell with `aChoices + bChoices < MIN_TRIALS_FOR_CONSISTENCY` (set to `2`) is excluded from the Trial Consistency calculation. The threshold is exposed as a const at the top of the resolver for easy adjustment.

**Rationale:** with only one trial, `max(proportionA, 1 - proportionA)` is always `1.0` regardless of underlying noise, so the metric is meaningless on single-trial cells. Setting `MIN_TRIALS_FOR_CONSISTENCY = 2` means we only count cells that actually had the chance to disagree with themselves.

**Verification:** unit test: cell with 1 decisive trial is excluded from `cellsObserved`. Cell with 2 decisive trials (1/1) IS included.

### A6c. Default drilldown selection (added after Codex plan M5)

**Decision:** the UI's default selected pair for the drilldown is the matrix row with the highest `meanAbsoluteDivergence` AMONG rows with `totalCells >= MIN_DRILLDOWN_DEFAULT_CELLS` (set to `10`). If no row meets the support threshold, the default is the row with the most `totalCells`. If the matrix is empty, no drilldown is rendered.

**Rationale:** without a support filter, a sparse pair with 1 wildly-divergent cell would be the default, which is the noisiest possible choice.

**Verification:** UI snapshot test: matrix with one high-divergence/low-support row + one moderate-divergence/high-support row → default selection is the high-support row.

### A7. No-overlap and empty-model handling

**Decision:**
- A model pair with zero cells in the comparison set returns a row with `totalCells: 0` and all metrics null. UI renders "no overlap."
- A model with zero cells anywhere in scope appears in `unavailableModels`, NOT in the matrix.

**Rationale:** the current resolver hard-fails when any selected model has no coverage. The new resolver tolerates this gracefully — a single empty model shouldn't blank the whole report. This is a deliberate, documented divergence from current behavior.

**Verification:** integration test with one fully-empty model + two normal models asserts `unavailableModels` contains the empty model and the matrix has 1 row (the two normal models).
