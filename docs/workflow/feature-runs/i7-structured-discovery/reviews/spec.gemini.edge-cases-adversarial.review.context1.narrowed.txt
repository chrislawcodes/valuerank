00041: ## Open Backlog
00042: 
00043: ### Ordering rationale
00044: 
00045: The four main items are sequenced deliberately:
00046: 
00047: **I-7 before I-9** (discovery before deterministic validation): if the spec is wrong,
00048: deterministic validation doesn't help — it just confirms the wrong thing was built correctly.
00049: The cascade effect of bad framing makes discovery the higher-leverage fix. Also cheaper.
00050: 
00051: **I-8 second** (handoff after discovery): a structured block note is essentially a snapshot
00052: of the discovery state at handoff time. The two items share the same data model — structured
00053: questions, open decisions, and acceptance criteria. Building discovery first makes handoff
00054: state nearly free.
00055: 
00056: **I-9 third** (validation after the above): requires design decisions about how the runner
00057: invokes test/build/lint for arbitrary repos, how it handles flaky tests, and whether
00058: validation is a new stage or a gate inside existing ones. Worth doing after the process
00059: gaps are closed.
00060: 
00061: **I-10 last** (modularize after the interface stabilizes): I-7, I-8, and I-9 all touch the
00062: runner. Modularizing first means touching the code twice. Do it once the shape of the
00063: interface is clear.

00067: ### I-7: Structured discovery state
00068: **Status:** Planned
00069: **Priority:** 1
00070: **Source:** 2026-03-22 adversarial evaluation (Codex + Claude)
00071: **What:** Replace the current free-text discovery fields with structured, machine-readable
00072: state in `state.json`:
00073: - `questions[]` — asked by the agent before spec
00074: - `answers{}` — recorded responses (keyed by question)
00075: - `assumptions[]` — explicit assumptions the spec rests on
00076: - `non_goals[]` — what is explicitly out of scope
00077: - `acceptance_criteria[]` — what done looks like
00078: - `unresolved[]` — open items that have not been answered
00079: 
00080: Make `unresolved` a first-class status: if any item is unresolved, `status` surfaces it
00081: prominently and `spec` is blocked from progressing until the human clears or defers each one.
00082: 
00083: **Why:** The pipeline currently validates artifact consistency, not product intent. A wrong
00084: spec sails through every gate. This is where wrong-product failures are born. Discovery is
00085: already mandatory (I-1) — this makes it enforceable.
00086: 
00087: **Why this order:** Cheapest fix with the highest cascade value. Also unlocks I-8 for nearly
00088: free.
00089: 
00090: **Cost:** Medium — runner changes to `command_discovery`, `command_status`, `command_spec`
00091: (add unresolved gate); `factory_state.py` schema additions; SKILL.md update; tests.

00240: ## Feedback Log
00241: 
00242: ### 2026-03-22 — Three-way adversarial evaluation (Codex → Claude → Codex)
00243: 
00244: **Diagnosis:** The pipeline catches process failures better than product failures. It is
00245: strongest at resumability, checkpointing, review traceability, and delivery bookkeeping.
00246: It is weakest at catching wrong framing, forcing real understanding before implementation,
00247: keeping the execution surface small, and validating truth outside the artifact chain.
00248: 
00249: **Key findings:**
00250: - False confidence from AI consensus: agents agreeing is not independent validation when
00251:   they read the same context with the same framing
00252: - Deterministic validation is absent: the entire pipeline relies on AI judgment
00253: - Discovery is the highest-value gate but not enforceable — unresolved items don't block
00254: - Session/context fragmentation: no model of what was known at handoff; wrong-feature
00255:   derailments are possible and have occurred
00256: - The runner is 2000+ lines; state mutation, orchestration, and formatting are entangled
00257: - Diff review degrades to theater once artifact size exceeds honest review capacity
00258: 
00259: **What the evaluation did not change:**
00260: - I-6 (lightweight mode) remains deferred — if anything, the case for it weakened
00261: - The existing stage structure is sound; the pipeline needs strengthening, not redesigning
00262: 
00263: ### 2026-03-20 — Codex retro + human observation
00264: 
00265: **What worked:**
00266: - Kept changes organized from spec → implementation → delivery
00267: - Pushed toward small reviewable slices instead of one large patch
00268: - Checkpoint discipline made the work feel more reliable
00269: 
00270: **What didn't:**
00271: - Runner fragility: `status` failed due to missing `workflow_utils` import (→ I-5)
00272: - Ceremony vs. value mismatch: felt heavier than warranted for semantic/UI work (→ I-6)
00273: - Branch staleness not surfaced before PR creation → merge conflicts (→ I-2, shipped)
00274: - Source-of-truth ambiguity: unclear which workflow files were authoritative (→ I-3, shipped)
00275: - No upfront requirements clarification (→ I-1, shipped)
