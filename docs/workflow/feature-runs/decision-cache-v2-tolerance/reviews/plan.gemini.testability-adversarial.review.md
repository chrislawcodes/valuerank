# Review: plan testability-adversarial

## Findings

| Severity | Finding | Evidence |
| --- | --- | --- |
| **HIGH** | The migration plan will break the web UI. The plan states the migration will remove `decisionCode` and `decisionCodeSource` from the `summaryCache` in the database. However, `decision-model-types.ts` defines `DECISION_MODEL_READ_RULES` which specify that the `web` surface uses `defaultMode: 'v1'` with `fallbackLayer: 'none'`. This means the web client is expected to handle v1 data directly and has no server-side mechanism to fall back on if the `decisionCode` it expects is missing. Deferring the consumer update to a subsequent PR will cause a production failure on analysis pages that read this cache. | `[CODE-CONFIRMED]` |
| **MEDIUM** | The plan introduces a logically inconsistent state by making the validator accept `{cacheVersion: 1, decisionState: "refusal"}`. The type `CachedWinnerFirstDecision` and the associated validator `isCachedWinnerFirstDecision` in the provided code show that `refusal` is not a valid state for `cacheVersion: 1`. The plan incorrectly labels this as "backward-tolerant." In reality, it's back-porting a new feature (`refusal`) to an old data version tag, creating a state that should not exist and was never generated by the v1 write path. This makes the type system and validators less reliable. | `[CODE-CONFIRMED]` |
| **MEDIUM** | The migration promotes all `decisionCode` values to "canonical" status without regard for their original derivation quality. The `summarize.py` worker uses a complex cascade of heuristics to find a `decisionCode`, and the resulting `parseClass` can be `exact`, `fallback_resolved`, or `ambiguous`. The migration plan makes no mention of checking the `parseClass` or `parsePath` from the existing `decisionMetadata`. It will therefore treat a low-confidence, ambiguously parsed `decisionCode` with the same authority as a high-confidence, exact match, potentially baking old parser bugs into the new canonical format permanently. | `[UNVERIFIED]` |
| **LOW** | The test matrix for the core derivation logic (`canonicalFromDecisionCode`) is incomplete. It covers the specified truth table but omits testing for unexpected string inputs for `decisionCode`. The python worker could have bugs or produce values outside the expected set (`"1".."5"`, `"refusal"`, `"other"`). The derivation helper should be tested to ensure it handles arbitrary string values gracefully (e.g., by mapping them to a `'unknown'` state) instead of crashing or producing an invalid object. | `[UNVERIFIED]` |

## Residual Risks

| Risk | Mitigation / Comment |
| --- | --- |
| **Loss of `refusal` signals for old data.** | The plan correctly notes that `refusal` signals on rows without a `decisionCode` will be lost, as there's no plan for retroactive inference. This is an accepted limitation but should be explicitly communicated, as it may affect longitudinal analysis of model behavior if old runs appear to have fewer refusals than they actually did. |
| **Migration script performance.** | The plan assumes the migration of ~280k rows will be reasonably fast. While idempotency allows for restarts, a very slow script could prolong the period where v1 and v2 cache items coexist, increasing the chance of v1 items being written by the live summarizer mid-migration. The `check-all-summarized` logic in `summarize-persistence.ts` does not appear to have been updated to handle the new cache version, which could lead to race conditions if the migration is slow. |
