00195: ## Data Flow Examples
00196: 
00197: ### Starting a Run
00198: 
00199: ```
00200: ┌────────┐     ┌─────────┐     ┌──────────┐     ┌─────────┐
00201: │ Web UI │────▶│ GraphQL │────▶│ Database │────▶│ PgBoss  │
00202: └────────┘     │ Mutation│     │ Create   │     │ Enqueue │
00203:                └─────────┘     │ Run      │     │ Jobs    │
00204:                                └──────────┘     └────┬────┘
00205:                                                      │
00206:                                                      ▼
00207:                                ┌──────────┐     ┌─────────┐
00208:                                │ Update   │◀────│ Worker  │
00209:                                │ Progress │     │ Process │
00210:                                └──────────┘     └─────────┘
00211: ```
00212: 
00213: 1. User clicks "Start Run" in web UI
00214: 2. `startRun` mutation creates Run record with PENDING status
00215: 3. Mutation enqueues `probe_scenario` jobs for each model×scenario pair
00216: 4. PgBoss workers pick up jobs, spawn Python probe.py
00217: 5. Python worker calls LLM provider, returns transcript
00218: 6. Orchestrator updates Transcript and Run.progress
00219: 7. When all probes complete, status changes to SUMMARIZING
00220: 8. Summarize jobs run, then status becomes COMPLETED
00221: 9. Analysis is auto-triggered
00222: 
00223: ### Querying Data via MCP
00224: 
00225: ```
00226: ┌────────────┐     ┌────────────┐     ┌─────────┐
00227: │ Local LLM  │────▶│ MCP Server │────▶│ GraphQL │
00228: │ (Claude)   │     │ (stdio)    │     │ Query   │
00229: └────────────┘     └────────────┘     └────┬────┘
00230:                                            │
00231:                                            ▼
00232:                    ┌────────────┐     ┌─────────┐
00233:                    │ Format for │◀────│ Database│
00234:                    │ Token Size │     │ Query   │
00235:                    └────────────┘     └─────────┘
00236: ```
00237: 
00238: 1. User asks Claude about ValueRank data
00239: 2. Claude calls MCP tool (e.g., `list_runs`)
00240: 3. MCP server authenticates via API key
00241: 4. Server executes GraphQL query
00242: 5. Results are formatted for token efficiency (<5KB)
00243: 6. Claude receives structured response

00247: ## Key Design Decisions
00248: 
00249: ### GraphQL over REST
00250: 
00251: **Why:** LLMs can introspect the schema and construct precise queries. Single endpoint simplifies auth and MCP integration. Flexible data fetching critical for token budgets.
00252: 
00253: ### PgBoss over Redis
00254: 
00255: **Why:** Uses same PostgreSQL database - no additional infrastructure. Built-in retry, priority queues, scheduling. Transactional with application data.
00256: 
00257: ### TypeScript Orchestrator + Python Workers
00258: 
00259: **Why:** Best of both worlds. TypeScript handles PgBoss natively. Python workers reuse LLM adapters from CLI tool. JSON stdin/stdout is simple and debuggable.
00260: 
00261: ### Single Tenant Architecture
00262: 
00263: **Why:** Internal team tool. All users share one workspace, all data visible to all users. No tenant_id columns, no ACLs needed.
00264: 
00265: ### JSONB for Flexible Schema
00266: 
00267: **Why:** Definition content structure varies by scenario type. JSONB provides schema flexibility without migrations. `schema_version` field enables future migrations at read time.
