## Findings
- HIGH [CODE-CONFIRMED] The plan leaves transcript side assignment dependent on host locale. It explicitly punts the `localeCompare` dependency as out of scope ([docs/workflow/feature-runs/unified-net-weighted-condition-score/plan.md:168-174]), but the shared helper still uses `favoredValueKey.localeCompare(opposedValueKey) < 0` to decide which side is “first” ([cloud/apps/web/src/utils/canonicalConditionSummary.ts:35-44]). That means the new `netScore` and direction can still vary by browser/OS locale for the same transcript set, so the transcript-based views are not actually deterministic under the new architecture.

- MEDIUM [CODE-CONFIRMED] The “single source of truth” boundary is only enforced by convention, not by architecture. The plan keeps `getCanonicalConditionBackground` and `getCanonicalConditionTextColor` exported and relies on grep gates to discourage direct use ([docs/workflow/feature-runs/unified-net-weighted-condition-score/plan.md:54-56], [docs/workflow/feature-runs/unified-net-weighted-condition-score/plan.md:142-154]). The current code already shows those raw helpers are public and called directly from both transcript tables ([cloud/apps/web/src/utils/canonicalConditionSummary.ts:144-155], [cloud/apps/web/src/components/analysis/ConditionDecisionsTable.tsx:626-639], [cloud/apps/web/src/components/analysis/PivotAnalysisTable.tsx:381-390]). A future caller can bypass `getConditionCellDisplay` without any compile-time barrier, so the plan does not truly lock the architecture to one render path.

## Residual Risks
- The plan still depends on process checks, not module boundaries, so any future score cell added outside the scoped files can reintroduce drift unless the grep gates are kept current and rerun consistently.
