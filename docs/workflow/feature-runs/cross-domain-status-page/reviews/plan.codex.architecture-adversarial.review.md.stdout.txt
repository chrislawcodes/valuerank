## Findings
- **High:** The reprobe flow is destructive before it is durable. The plan soft-deletes the current transcript, then enqueues the new `probe_scenario` job after the transaction commits. If `boss.send()` fails, or if the worker never runs, the slot is left without a live transcript and without a guaranteed replacement probe. Because the detector only scans non-deleted transcripts, the anomaly can disappear from the status surface even though the slot is still broken.
- **High:** The concurrency guard does not actually prevent duplicate reprobes. `SELECT ... FOR UPDATE` serializes the DB row update, but the pending-job check is outside the transaction and the enqueue happens after commit. Two clicks can both pass the “no pending job” check before either queue write is visible, which can produce two probe jobs for the same slot and double-delete the same transcript state.
- **Medium [UNVERIFIED]:** The “max 3 re-probes” circuit breaker counts soft-deleted transcripts as the retry ledger, but the plan assumes every soft delete for that slot comes from this reprobe path. If the existing codebase has any other soft-delete path, import cleanup, or historical data shape that also uses `deleted_at`, the limit will miscount and either block legitimate retries early or fail to block the real retry loop.

## Residual Risks
- The new `/status` page polls every 5 seconds and the anomaly query is unpaginated. That is fine for the current scale, but it will get expensive quickly if open anomalies grow.
- `displaySubject` depends on a delimiter-based slot tuple. If any identifier can contain the delimiter, or if slot-keyed anomaly types expand later, the formatter will need a stronger encoding than string parsing.
- `resolveRunAnomaly` is a dismissal, not a repair. If the underlying empty-response condition still exists, the next sweep will recreate the anomaly. That behavior needs to be obvious in the UI copy so operators do not treat it like a permanent fix.
- The plan assumes the current flat-user model stays in place. If per-domain access is added later, both mutations will need an authorization pass and the current contract will be wrong.
