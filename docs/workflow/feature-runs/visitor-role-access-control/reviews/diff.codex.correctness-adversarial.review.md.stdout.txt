## Findings
- High: `cloud/apps/api/src/auth/services.ts` and `cloud/apps/api/src/auth/middleware.ts` now fail open on missing `role`. `signToken()` stamps absent roles as `ADMIN`, and JWT verification does the same when reading `payload.role`. That means any token minted from a partial user object, or any token that lacks the new claim, is treated as admin instead of being rejected or downgraded.
- High: `cloud/apps/api/src/graphql/index.ts` calls `getOperationAST(document)` without passing the selected `operationName`. For documents that contain multiple operations, GraphQL.js returns `null` unless the operation name is supplied, so the admin gate can be skipped on named mutation requests.
- Medium [UNVERIFIED]: `cloud/apps/api/src/graphql/index.ts` now blocks every mutation for non-admins. If the existing schema still has any unauthenticated or visitor-scoped mutations, this change will break them outright.

## Residual Risks
- I did not verify whether the app currently relies on any legacy JWTs or partial token payloads, so the practical blast radius of the fail-open role default is uncertain.
- I did not verify the full GraphQL schema, so it is unclear whether the blanket mutation gate is compatible with all existing mutation flows.
