## Findings
- High [CODE-CONFIRMED]: The spec contradicts itself on the dedupe key for condition counts. One definition says a filled condition is `DISTINCT (runId, scenarioId, modelId, sampleIndex)`, but the high-priority invariant later says `pairedConditionCount` and `orphanedConditionCount` must dedupe on `(scenarioId, modelId, sampleIndex)` across runs. That slot-based rule matches the existing completeness logic in [coverage-completeness.ts](/Users/chrislaw/valuerank/.claude/worktrees/infallible-bassi-fafa71/cloud/apps/api/src/services/run/coverage-completeness.ts) and the fact that [Transcript](/Users/chrislaw/valuerank/.claude/worktrees/infallible-bassi-fafa71/cloud/packages/db/prisma/schema.prisma) has no uniqueness on the slot tuple. If the implementation follows the run-scoped version, retries will inflate counts and the gap gate will fire when it should not.

- Medium [CODE-CONFIRMED]: The launch preview is under-specified for cells that aggregate multiple companion definitions. [buildRunJobPlan()](/Users/chrislaw/valuerank/.claude/worktrees/infallible-bassi-fafa71/cloud/apps/api/src/services/run/start-plan.ts) computes the job plan from one definition’s `definitionScenarioIds`, not a cell-level union, so the card must know which exact definition is being launched before it can compute the correct `scenarioCount` and after-state. The spec only says the route state should carry “all definitionIds contributing to that cell”, which is not enough to derive the right launch math when those definitions do not share the same scenario set.

- Low [CODE-CONFIRMED]: The incomplete-batch warning text is written as a singular example, but the resolver already accumulates a count across runs/definitions for the cell. In [domain-coverage.ts](/Users/chrislaw/valuerank/.claude/worktrees/infallible-bassi-fafa71/cloud/apps/api/src/graphql/queries/domain-coverage.ts), `incompleteBatchCount` is summed, so the real value can be greater than 1. As written, the warning copy will be inaccurate for any cell with more than one incomplete batch.

## Residual Risks
- The spec still does not pin the exact GraphQL field names or returned shape for the new per-direction condition data, so implementation can drift unless the plan locks that schema down.
- The new single-direction launch mode and the popover/card wiring are not present in the provided code, so the refresh/new-tab fallback and the exact `definitionId` versus lagging-direction validation remain unverified.
- The spec still leaves the definition-selection rule open when an aggregated cell spans definitions with different scenario counts, so that edge case needs an explicit test even after the launch contract is finalized.
